# Draft Cup — Specification

## What is Draft Cup?

Draft Cup is a web platform for running competitive team draft tournaments for Mount & Blade II: Bannerlord multiplayer. The core concept is an **auction-based team formation system**: tournament organizers run live auctions in which team captains bid against each other to acquire players for their roster. This creates balanced, community-driven teams with an element of strategy around budgeting, player valuation, and class composition.

The platform covers the full lifecycle of a tournament: player pool setup, auction configuration, live bidding sessions, group-stage match tracking, bracket management, and a finals pick-ban process for map and faction selection.

---

## Tech Stack

**Backend**: Bun runtime, Hono web framework, SQLite via `bun:sqlite`, JWT-based authentication via HTTP-only cookies.

**Frontend**: React 19, Vite 7, TypeScript (strict, erasable syntax only), React Router v7, Tailwind CSS v4, shadcn/ui components, Lucide icons. Fonts: Barlow Condensed (display/headings), DM Sans (body), JetBrains Mono (monospace). Theme: dark-only, zinc-950 base, amber-500 accent.

---

## Roles & Authentication

Four distinct roles, each with different access scopes:

**Admin** — Single superuser authenticated by an environment variable password (`ADMIN_PASSWORD`). No database row. Issues a short-lived JWT on login. Can manage everything: create/delete host accounts, set featured tournaments, manage tokens, override scores, reset brackets, manage the global map pool.

**Host** — Tournament organizer. Stored in the `hosts` table with a bcrypt-hashed password and a session token. Each host can own multiple tournaments. Hosts can manage their own tournaments but cannot access others'. A host's session token is single-instance: a new login revokes the previous session.

**Auctioneer** — Volunteer who runs the live bidding screen. Authenticated by a short token (`DRAFT-XXXXXX`) generated per-tournament and stored on the tournament row. One token covers all auctions in a tournament since divisions run sequentially. Multiple auctioneers can share the token.

**Captain** — Team captain participating in an auction. Authenticated by a per-captain token generated by the host. Scoped to a single auction. Can bid, rename their team, and participate in the pick-ban session if they reach the finals.

Authentication is cookie-based (HTTP-only, `SameSite=Strict`). The middleware (`src/middleware/auth.ts`) validates JWTs on every request and attaches the decoded payload to the request context via Hono's typed environment.

## Database Schema (v10)

### `hosts`
Tournament organizers. Fields: `id`, `username` (unique), `password_hash`, `token` (session token, null = logged out), `token_generated_at`, `created_at`.

### `tournaments`
Top-level tournament container, e.g. "Draft Cup XV". Belongs to a host. Fields: `id`, `name`, `slug` (unique, URL-safe, auto-generated from name), `description`, `host_token`, `auctioneer_token` (shared across all auctions), `map_pool` (JSON array of map IDs for rolling), `finals_map_pool` (JSON array of map IDs for pick-ban), `players_per_team` (default 6, excluding the captain), `status` (active/archived), `is_featured` (at most one tournament is featured globally), `created_at`.

### `players`
The player pool for a tournament, shared across all auctions within it. Fields: `id`, `tournament_id`, `name`, `classes` (comma-separated string: `inf`, `arc`, `cav` — a player can have multiple), `is_available` (set to 0 when purchased), `created_at`.

### `auctions`
One per division within a tournament (e.g. "Division A", "Division B"). Fields: `id`, `tournament_id`, `name`, `min_increment` (default 0.1), `bid_cooldown_seconds` (pause after each bid before closing, default 3), `is_public`, `status` (setup → ready → finished), `created_at`.

There is **no default budget** at the auction level — every captain has an individually configured budget, set manually by the host to allow skill-based handicapping.

### `captains`
One row per team captain slot per auction. Expected to be exactly 8 per auction. Fields: `id`, `auction_id`, `display_name`, `team_name` (set by captain after login), `token` (login code), `budget`, `class` (optional: inf/arc/cav — the captain's own class).

### `auction_sessions`
A live run of an auction. Fields: `id`, `auction_id`, `status` (pending → live → paused → finished), `sold_count`, `skipped_count`, `half_budget` (1/0, toggle controlling whether the 50%-budget cap is enforced), `created_at`, `started_at`, `finished_at`.

### `session_queue`
Ordered list of players up for auction within a session. Fields: `id`, `session_id`, `player_id` (nullable), `player_name` (snapshot), `queue_position`, `status` (pending → active → sold). Unique on `(session_id, queue_position)`.

### `session_purchases`
Record of every successful sale. Fields: `id`, `session_id`, `captain_id`, `player_id`, `player_name` (snapshot), `price`, `purchased_at`.

### `session_bids`
All bids placed during a session. The most-recent row per `queue_entry_id` is the current top bid. Fields: `id`, `session_id`, `queue_entry_id`, `captain_id`, `captain_name`, `amount`, `placed_at`.

### `session_chat`
Free-text chat messages within a session. Fields: `id`, `session_id`, `author_role`, `author_name`, `content`, `sent_at`.

### `factions`
Global static table, seeded once at DB init. The 6 Bannerlord factions: Vlandia, Battania, Sturgia, Aserai, Khuzait, Empire.

### `maps`
Global map pool, managed exclusively by the admin (not per-tournament). Fields: `id`, `name`, `tags`, `game_id` (in-game identifier used in server commands, e.g. `mp_skirmish_valley`; optional, falls back to name if null), `image_path` (uploaded image), `is_active`. Tournaments reference maps by ID in their `map_pool` and `finals_map_pool` JSON columns.

### `matchups`
One rolled result per match. Captures which map is played and which faction each side uses. Fields: `id`, `tournament_id`, `bracket_id` (null for shared/non-bracket matchups), `round`, `label` (e.g. "A:1v2", "SF1", "Final"), `map_id`, `faction_a_id`, `faction_b_id`, `is_public` (0 = draft; 1 = visible to all), `created_at`.

Matchups are rolled one at a time by the host. The roller picks a random available map (excluding already-used or manually excluded maps in the current round) and assigns two random factions.

### `brackets`
One per auction. Represents the tournament bracket for a division. Fields: `id`, `tournament_id`, `auction_id`, `name`, `slots` (JSON array: captain assignments to the 8 slots), `locked` (0 = host can shuffle; 1 = locked, only admin can reset), `is_public`, `created_at`.

### `matches`
Individual matches within a bracket. Generated when the bracket is first generated. Fields: `id`, `bracket_id`, `round` (1–3 = group stage, 4 = semi-finals, 5 = finals), `match_order`, `match_label`, `group_label` ('A', 'B', or null for knockouts), `captain_a_id`, `captain_b_id`, `matchup_id`, `score_a`, `score_b`, `winner_captain_id`, `status` (pending/played), `is_finals` (1 = triggers pick-ban creation when both finalists are known).

### `pick_ban_sessions`
One per finals match. Created by the host once both finalists are determined. Fields: `id`, `bracket_id`, `match_id` (unique), `status` (waiting → banning → picking → complete), `captain_a_id`, `captain_b_id`, `map_pool` (JSON array of 5 map IDs selected by host), `ban_sequence` (JSON, default `["a","b","b","a"]`), `ban_turn`, `bans` (JSON array of completed bans), `a_joined`/`b_joined`, `chosen_map_id` (the surviving map after all bans), `a_pick`/`b_pick` (faction IDs, hidden until revealed), `revealed`, `created_at`, `completed_at`.

## Tournament Format

Every tournament uses a fixed 8-team format:

**Group Stage** — 8 teams split into Group A (teams 1–4) and Group B (teams 5–8). Each group plays a round-robin over 3 rounds: 2 matches per group per round, 6 matches per group, 12 total group-stage matches.

**Semi-Finals** — 1st of Group A vs 2nd of Group B, and 2nd of Group A vs 1st of Group B. Two matches.

**Finals** — The two semi-final winners face each other. This match triggers the pick-ban session.

**Bracket generation** — The host provides an ordered list of captain IDs. The system assigns them to group slots, generates all round-robin match pairings, and creates semi-final and final placeholder rows.

**Matchup rolling** — For each match the host rolls one matchup (map + two factions) via the roller UI. Maps already used in published matchups in the current round are automatically excluded. The host can additionally exclude specific maps manually. A published matchup cannot be re-rolled.

---

## Pick-Ban Process (Finals Only)

The host creates a pick-ban session for the finals match once both finalists are known. The host selects 5 maps from the tournament's `finals_map_pool` for that session.

**Ban phase** — Both captains log in to `/pickban/:sessionId`. The ban sequence is A → B → B → A (4 bans total), leaving exactly one map. Each captain confirms their identity by joining the session; banning begins once both have joined.

**Pick phase** — Each captain secretly picks a faction from the 6 available. Picks are hidden until both captains have submitted.

**Reveal** — Once both picks are in, the results are revealed simultaneously. The system generates the finals matchup row with the surviving map and the two chosen factions.

**Command** — The result is displayed as a `!setmap` server command: `!setmap <game_id|name> <faction_a> <faction_b>`, with faction names lowercased. The `game_id` field on the map is used if set; otherwise the map name is used as fallback. A copy-to-clipboard button is provided.

## Map Management

Maps are managed globally by the admin via the Maps page (`/admin/maps`). They are not per-tournament.

Each map has:
- **name** — Display name
- **game_id** — In-game identifier used in server commands (e.g. `mp_skirmish_valley`). Falls back to `name` if not set.
- **image** — Uploaded image shown as background in matchup cards
- **tags** — Optional tags for organization

Each tournament has a `map_pool` (JSON array of map IDs used for rolling group-stage / semi-final matchups) and a `finals_map_pool` (JSON array of map IDs available for the host to pick the pick-ban map pool from). These are configured in the tournament's Config tab.

## Frontend Architecture

### Pages

**Public** (no auth):
- `/` — Home page. Lists tournaments.
- `/t/:slug` — Public tournament page: bracket visualization, published matchups, live session links, team roster links.
- `/archive` — List of all tournaments.

**Login**:
- `/login` — Single tabbed login covering host, auctioneer, and captain. Each uses a different credential type.
- `/admin-login` — Separate admin login page.

**Host** (`/host/projects/:slug`):
- Narrow icon sidebar (w-16) showing one icon per tournament the host owns. Clicking navigates to that tournament's detail view.
- On load with no slug: if the host owns the featured tournament, navigate to it. Otherwise show a prompt.
- `EventBubble` notification widget active.

**Admin** (`/admin`):
- 240px sidebar listing hosts (with online/offline dot) and their tournaments indented beneath. Context menu on right-click for management actions.
- Clicking a tournament opens `TournamentDetail` in the right pane (admin roleOverride).
- Map management accessible via Maps icon in sidebar.

**Auctioneer** (`/auctioneer`):
- Narrow icon sidebar with single tournament icon.
- Renders `TournamentDetail` with `roleOverride="auctioneer"`.

**Captain** (`/captain`):
- Renders `TournamentDetail` with `roleOverride="captain"` (read-only view of bracket, matchups, etc.).

**Pick-Ban** (`/pickban/:sessionId`):
- Interactive map ban and faction pick UI for the two finalists.
- Shows joining status, ban sequence with map grid, faction picker (circular icons), and on completion displays `FinalsMatchupCard` with the `!setmap` command.
- "Back to dashboard" link on the complete screen.

**Live Auction** (`/t/:slug/auction/:sessionId`):
- Public/shared live auction viewer. Shows current player, bid history, upcoming queue, captain rosters. Uses WebSocket for real-time updates.

### Shared `TournamentDetail` Component

Core of the host/admin/auctioneer/captain experience (`client/src/components/tournament/TournamentDetail.tsx`). Takes a `slug` and optional `roleOverride`. Renders a tabbed interface:

**Tab: Player Pool** — Searchable, filterable player table. Inline editing and bulk import (JSON/CSV). Class filter pills (Inf/Arc/Cav). Non-managers (auctioneers, captains) see the table read-only.

**Tab: Matchups** — Round selector, map grid (already-used maps auto-excluded or crossed out), rolling animation, result display. List of all rolled matchups below with publish controls.

**Tab: Config** — Auctioneer token management, map pool editor (select which global maps to include), finals map pool editor, tournament description.

**Auction Tabs** (one per auction) — Captain slots table (name, budget, token), bracket creation and management, bracket shuffle/generate.

## Key Design Decisions

**No polling** — Pages fetch data once on load. Real-time updates use WebSocket exclusively: the public auction viewer (`AuctionSessionPage`) and the pick-ban session (`PickBanPage`, `PickBanModal`) both connect to dedicated WS endpoints (`/ws/auction/:id`, `/ws/pickban/:id`). The backend broadcasts a lightweight update message after any state change, and clients re-fetch the full state in response.

**Single featured tournament** — The admin sets exactly one tournament as featured. Setting a new one automatically unfeatures the previous.

**Auctioneer token is per-tournament** — One token covers all auctions in a tournament. Since divisions run sequentially, the same auctioneer (or group sharing the token) operates all sessions.

**Captain budgets are always manual** — No default budget. Every captain's budget is set individually by the host to allow skill handicapping.

**Half-budget rule is a session toggle** — The restriction that a captain cannot spend more than half their budget on a single player is toggled per session (`half_budget` flag on `auction_sessions`), not hardcoded.

**Players have a `classes` field, not a hierarchy** — A player can play infantry, archer, cavalry, or any combination. Stored as comma-separated `inf`/`arc`/`cav`. No primary class.

**Brackets are fixed 8-team format** — 2 groups of 4, round-robin, semis, finals. Hardcoded structure. The only variable is which captain occupies which slot.

**Pick-ban is finals-only** — Group stage and semi-final maps are assigned by the matchup roller. Only the finals match gets the interactive pick-ban session.

**Scores are entered manually** — Hosts and admins enter match scores directly in the bracket UI. No screenshot upload or automated verification.

**Maps are global, admin-managed** — The map pool is not per-tournament. The admin maintains a single global list. Each tournament references a subset of those maps via its `map_pool` and `finals_map_pool` ID arrays.

**`game_id` for server commands** — Each map has an optional `game_id` field storing the in-game identifier (e.g. `mp_skirmish_valley`). This value is used in `!setmap` commands. Falls back to the map's display name if not set.
